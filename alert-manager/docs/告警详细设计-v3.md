# 告警管理系统详细设计 v3

## 二、字段默认值说明

### 2.1 时间字段默认值约定

| 默认值 | 含义 | 使用场景 |
|--------|------|----------|
| `CURRENT_TIMESTAMP` | 当前时间 | 创建时间、更新时间 |
| `'1970-01-01 00:00:00'` | 未设置 | 确认时间、清除时间、处置时间、同步时间、发送时间 |
| `'9999-12-31 23:59:59'` | 永久有效 | 结束时间（如果需要） |

### 2.2 数值字段默认值约定

| 默认值 | 含义 | 使用场景 |
|--------|------|----------|
| `0` | 未设置/未关联 | 外键ID、计数器、金额 |
| `1` | 默认启用 | 状态字段、触发次数 |

### 2.3 字符串字段默认值约定

| 默认值 | 含义 | 使用场景 |
|--------|------|----------|
| `''` | 空字符串 | 可选的描述、备注、名称等字段 |
| `'ALWAYS'` | 默认值 | 启动规则类型 |
| `'ACTIVE'` | 默认状态 | 告警状态 |
| `'PENDING'` | 待处理 | 同步状态、发送状态 |

### 2.4 JSON 字段默认值约定

| 默认值 | 含义 | 使用场景 |
|--------|------|----------|
| `(JSON_OBJECT())` | 空对象 `{}` | 配置JSON、消息JSON |
| `(JSON_ARRAY())` | 空数组 `[]` | 列表JSON、图片URL列表 |

## 三、业务逻辑中的默认值判断

### 3.1 时间字段判断

```java
// 判断是否已确认
if (alarm.getAcknowledgedTime().isAfter(LocalDateTime.of(1970, 1, 1, 0, 0, 0))) {
    // 已确认
}

// 或使用工具方法
public static boolean isTimeSet(LocalDateTime time) {
    return time != null && time.isAfter(LocalDateTime.of(1970, 1, 1, 0, 0, 0));
}
```

### 3.2 外键字段判断

```java
// 判断是否已委托
if (alarm.getAssigneeId() > 0) {

// 判断是否关联产品
if (alarm.getProductId() > 0) {
    // 已关联产品
}
```

### 3.3 字符串字段判断

```java
// 判断是否有备注
if (StringUtils.isNotBlank(alarm.getAcknowledgedRemark())) {
    // 有备注
```

## 四、数据库设计优势

### 4.1 避免 NULL 值的优势

1. **查询性能提升**：避免 NULL 值判断，索引效率更高
2. **代码简化**：不需要频繁进行 NULL 检查
3. **数据一致性**：统一使用默认值，避免 NULL 和空值混用
4. **存储优化**：MySQL 对 NOT NULL 字段有更好的存储优化

### 4.2 默认值的优势

1. **业务语义清晰**：通过默认值表达业务含义
2. **减少错误**：避免因 NULL 值导致的 NPE
3. **简化查询**：不需要 `IS NULL` 或 `IS NOT NULL` 判断
4. **兼容性好**：与 MyBatis-Plus 等 ORM 框架配合更好

## 五、注意事项

### 5.1 device_count 冗余字段维护

`alarm_scheme.device_count` 字段记录告警方案关联的设备数量，需要在以下场景更新：

#### 更新时机

1. **创建告警方案后**
   - 查询测站关联的设备
   - 为每个设备创建 ThingsBoard 规则
   - 统计设备数量并更新 `device_count`

2. **更新告警方案（测站关联变化）**
   - 新增测站：查询新测站的设备，增加设备数量
   - 移除测站：删除对应设备的规则，减少设备数量
   - 重新统计并更新 `device_count`

3. **测站-设备关联变化（外部系统触发）**
   - 监听外部系统的设备关联变化事件
   - 重新统计并更新 `device_count`

4. **定时同步任务**
   - 定期（如每天凌晨）重新统计所有告警方案的设备数量
   - 修正可能的数据不一致

#### 统计逻辑

```sql
-- 统计单个告警方案的设备数量
UPDATE alarm_scheme 
SET device_count = (
  SELECT COUNT(DISTINCT device_id) 
  FROM alarm_scheme_device_rule 
  WHERE scheme_id = ? AND sync_status = 'SUCCESS'
)
WHERE id = ?;

-- 批量统计所有告警方案的设备数量
UPDATE alarm_scheme s
SET device_count = (
  SELECT COUNT(DISTINCT device_id) 
  FROM alarm_scheme_device_rule r
  WHERE r.scheme_id = s.id AND r.sync_status = 'SUCCESS'
)
WHERE deleted = 0;
```

#### 注意事项

1. **只统计同步成功的设备**：`sync_status = 'SUCCESS'`
2. **使用 DISTINCT 去重**：避免重复统计
3. **事务一致性**：在事务中更新设备规则和 device_count
4. **异步更新**：对于大批量操作，可以异步更新 device_count

### 5.2 时间字段特殊值

使用 `'1970-01-01 00:00:00'` 表示未设置时，需要注意：
- 在业务逻辑中统一判断标准
- 在前端展示时，需要特殊处理（不显示或显示为"未设置"）
- 在统计分析时，需要排除这个特殊值

### 5.2 JSON 字段默认值

MySQL 8.0+ 支持 JSON 字段默认值：
```sql
config_json JSON NOT NULL DEFAULT (JSON_OBJECT())
```

如果 MySQL 版本 < 8.0，需要在应用层处理：
```java
@TableField(typeHandler = JacksonTypeHandler.class)
private JSONObject configJson = new JSONObject();
```

### 5.3 TEXT 字段

TEXT 类型字段不能设置默认值，需要在应用层处理：
```java
private String stepDescription = "";
```
