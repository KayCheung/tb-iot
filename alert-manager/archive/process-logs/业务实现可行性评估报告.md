## 二、核心业务流程可行性分析

### 2.1 告警方案创建与同步 ✅ 可行

#### 业务流程
```
1. 用户创建告警方案
   ├─ 配置指标规则（独立/联动）
   ├─ 配置级别条件（阈值、操作符、持续时间）
   ├─ 配置启动规则（ALWAYS/SCHEDULED/CUSTOM/CONDITION）
   └─ 关联测站

2. 系统查询测站关联的设备
   └─ 调用外部接口：GET /api/stations/{stationId}/devices

3. 为每个设备生成 tb_alarm_type
   └─ 格式：{scheme_code}_{device_id}

4. 生成 TBEL 脚本（包含启动规则）
   ├─ 启动规则判断代码
   ├─ 指标条件判断代码
   ├─ 持续时间判断代码
   └─ 返回布尔值

5. 构建 CalculatedField 配置
   ├─ arguments: 所有指标的数据源
   ├─ expression: TBEL 脚本字符串
   └─ output: 输出到属性，触发规则链

6. 调用 ThingsBoard API 创建 CalculatedField
   └─ POST /api/calculatedField

7. 保存映射关系到 alarm_scheme_device_rule 表
   ├─ tb_alarm_type（唯一标识）
   ├─ tb_calculated_field_id（ThingsBoard 返回的 UUID）
   └─ sync_status（同步状态）
```

#### 可行性分析
| 环节 | 可行性 | 说明 |
|------|--------|------|
| 测站设备查询 | ✅ 可行 | 通过外部接口获取 |
| tb_alarm_type 生成 | ✅ 可行 | 格式明确，全局唯一 |
| CalculatedField 构建 | ⚠️ 需验证 | 需要验证复杂表达式的支持程度 |
| API 调用 | ✅ 可行 | 接口明确，文档完整 |
| 映射关系存储 | ✅ 可行 | 表结构完整 |

#### 潜在问题
1. **表达式复杂度限制**
   - 问题：不确定 ThingsBoard 支持多复杂的表达式
   - 影响：如果表达式过于复杂可能失败
   - 解决方案：
     - 先测试简单表达式（单条件）
     - 逐步测试复杂表达式（多条件 OR/AND）
     - 如果不支持，拆分为多个 CalculatedField

2. **跨设备联动**
   - 问题：CalculatedField 的 `refDynamicSourceConfiguration` 支持关系查询
   - 影响：需要验证是否支持直接引用其他设备的指标
   - 解决方案：
     - 测试 `RelationPathQueryDynamicSourceConfiguration`
     - 如果不支持，联动逻辑在告警系统侧实现

### 2.2 告警触发与消息处理 ✅ 可行

#### 业务流程
```
1. ThingsBoard 计算字段触发
   └─ 指标值满足表达式条件

2. TBEL 脚本执行
   ├─ 启动规则判断（SCHEDULED/CUSTOM/CONDITION）
   ├─ 指标条件判断
   ├─ 持续时间判断
   └─ 返回 true/false

3. ThingsBoard 规则链发送 Kafka 消息
   └─ 消息包含 type 字段（tb_alarm_type）

4. 告警系统消费 Kafka 消息
   └─ 解析消息，提取 tb_alarm_type

5. 查询告警方案映射
   └─ SELECT * FROM alarm_scheme_device_rule WHERE tb_alarm_type = ?

6. 检查方案状态
   └─ 方案是否启用（status = 1）

7. 创建/更新告警实例
   ├─ 首次触发：创建新记录
   └─ 重复触发：更新触发次数和时间

8. 触发通知
   └─ 匹配通知规则，发送通知
```

#### 可行性分析
| 环节 | 可行性 | 说明 |
|------|--------|------|
| Kafka 消息接收 | ✅ 可行 | 消息格式明确 |
| tb_alarm_type 定位 | ✅ 可行 | 唯一索引，查询高效 |
| 启动规则判断 | ✅ 可行 | 在告警系统侧实现 |
| 告警实例创建 | ✅ 可行 | 表结构完整 |
| 通知触发 | ✅ 可行 | 规则匹配逻辑清晰 |

#### 关键设计点
1. **启动规则在告警系统侧实现**
   - ThingsBoard 只负责触发告警
   - 告警系统根据 `enable_rule_config` 进行二次过滤
   - 优点：灵活性高，不依赖 ThingsBoard 的时间规则能力

2. **持续时间判断在告警系统侧实现**
   - ThingsBoard 不支持持续时间判断
   - 告警系统需要记录首次触发时间
   - 判断逻辑：`当前时间 - 首次触发时间 >= duration_seconds`

### 2.3 告警处理与状态同步 ✅ 可行

#### 业务流程
```
1. 确认告警
   ├─ 更新本地状态：alarm_status = 'ACKNOWLEDGED'
   ├─ 调用 ThingsBoard API：POST /api/alarm/{alarmId}/ack
   └─ 记录处理日志

2. 委托告警
   ├─ 更新本地状态：assignee_id, assignee_name
   ├─ 同步到 ThingsBoard（如果支持）
   └─ 发送通知给委托人

3. 清除告警
   ├─ 更新本地状态：alarm_status = 'CLEARED'
   ├─ 调用 ThingsBoard API：POST /api/alarm/{alarmId}/clear
   └─ 记录处理日志

4. 处置告警
   ├─ 更新本地状态：alarm_status = 'RESOLVED'
   └─ 记录处理日志
```

#### 可行性分析
| 环节 | 可行性 | 说明 |
|------|--------|------|
| 确认告警 | ✅ 可行 | API 支持 |
| 清除告警 | ✅ 可行 | API 支持 |
| 委托告警 | ⚠️ 需确认 | 需确认 ThingsBoard 是否支持 assignee |
| 处置告警 | ✅ 可行 | 本地状态管理 |

### 2.4 通知管理 ✅ 可行

## 五、技术风险评估

### 5.1 高风险项 🟢 → 风险已消除

#### ~~风险 1：CalculatedField 表达式能力不足~~ ✅ 已解决
- **发现**：ThingsBoard 支持 TBEL (JavaScript) 脚本
- **能力**：完整的 JavaScript 语法支持
- **结论**：表达式能力无限制，风险消除

#### 风险 2：ThingsBoard API 限流 → 🟡 中风险
- **描述**：批量创建/更新时可能触发限流
- **影响**：同步失败，需要重试
- **概率**：中（40%）
- **缓解措施**：
  1. 实现限流控制（50 次/秒）
  2. 实现失败重试机制
  3. 使用线程池控制并发数

### 5.2 中风险项 🟢 → 风险已降低

#### ~~风险 3：跨设备联动不支持~~ ✅ 已解决
- **发现**：TBEL 脚本可以访问多个参数
- **能力**：通过配置不同设备的指标作为参数实现联动
- **结论**：跨设备联动可实现，风险消除

#### 风险 4：外部接口不稳定 → 🟡 中风险
- **描述**：设备、测站信息查询接口可能不稳定
- **影响**：无法获取设备信息，影响告警创建
- **概率**：中（30%）
- **缓解措施**：
  1. 实现缓存机制
  2. 实现降级策略
  3. 实现重试机制

### 5.3 低风险项 🟢

#### 风险 5：数据一致性问题
- **描述**：本地数据与 ThingsBoard 数据不一致
- **影响**：告警状态不准确
- **概率**：低（10%）
- **缓解措施**：
  1. 实现定时同步任务
  2. 实现状态监控
  3. 提供手动同步功能

```java
@Test
public void testCreateSimpleCalculatedField() {
    CalculatedField cf = new CalculatedField();
    cf.setEntityId(new EntityId("DEVICE", deviceId));
    cf.setType("SIMPLE");
    cf.setName("TEST_TEMP_001_3001");
    
    // 构建配置
    SimpleCalculatedFieldConfiguration config = new SimpleCalculatedFieldConfiguration();
    
    // 添加参数
    Map<String, Argument> arguments = new HashMap<>();
    Argument arg = new Argument();
    arg.setRefEntityKey(new ReferencedEntityKey("temperature", "TS_LATEST", "SERVER_SCOPE"));
    arg.setDefaultValue("0");
    arguments.put("temp", arg);
    config.setArguments(arguments);
    
    // 设置表达式
    config.setExpression("temp > 80");
    
    // 设置输出
    AttributesOutput output = new AttributesOutput();
    output.setName("TEST_TEMP_001_3001");
    output.setScope("SERVER_SCOPE");
    AttributesRuleChainOutputStrategy strategy = new AttributesRuleChainOutputStrategy();
    strategy.setType("RULE_CHAIN");
    output.setStrategy(strategy);
    config.setOutput(output);
    
    cf.setConfiguration(config);
    
    // 调用 API
    CalculatedField result = thingsBoardClient.saveCalculatedField(cf);
    
    assertNotNull(result.getId());
    System.out.println("Created CalculatedField ID: " + result.getId().getId());
}
```
