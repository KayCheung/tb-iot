## 主要变更

### 1. 表结构调整

#### 新增表
- `alarm_scheme_device_rule`: 告警方案设备规则映射表（核心表）
  - 存储 ThingsBoard 告警规则类型（tb_alarm_type）与告警方案的映射关系
  - 解决通过 ThingsBoard 消息反查告警方案的问题

- `alarm_instance_station`: 告警实例测站关联表
  - 采用关联表方式替代 JSON 存储
  - 便于查询和索引

#### 修改表
- `alarm_scheme`: 新增 `scheme_code` 字段（唯一编码）
- `alarm_scheme_metric_rule`: 新增 `device_id` 字段（支持联动其他设备）
- `alarm_instance`: 
  - 新增 `tb_alarm_type` 字段
  - 新增 `assignee_id/assignee_name` 字段（委托人）
  - 移除 `station_ids` JSON 字段
  - 移除 `acknowledged_by` 改为 `assignee_id`
- `alarm_handle_log`: 新增 `ASSIGN` 处理类型
- `notify_user_group`: 移除 `parent_id` 字段（扁平结构）

#### 删除表
- `station`: 通过外部接口获取
- `station_type`: 通过外部接口获取
- `station_device_rel`: 通过外部接口获取
- `device`: 通过外部接口获取
- `product`: 通过外部接口获取

### 2. 核心概念明确

#### ThingsBoard 告警规则类型（type）
- 格式：`{scheme_code}_{device_id}`
- 示例：`PUMP_TEMP_001_3001`
- 作用：作为唯一标识，用于反查告警方案

#### 联动触发实现
- 由 ThingsBoard 实现联动逻辑
- 一个告警规则可以同时设置多个指标（当前设备或其他设备）
- 告警管理系统只需记录配置，同步到 ThingsBoard

#### 启动规则处理
- 启动规则配置同步到 ThingsBoard
- 由 ThingsBoard 控制告警的启用时间
- 降低告警管理系统的复杂性

### 3. 业务流程优化

#### 告警方案创建流程
1. 创建告警方案（本地数据库）
2. 查询测站关联的设备（外部接口）
3. 为每个设备在 ThingsBoard 创建告警规则
4. 保存映射关系到 `alarm_scheme_device_rule` 表

#### 告警触发流程
1. ThingsBoard 触发告警，发送 Kafka 消息
2. 消息中包含 `type` 字段（告警规则类型）
3. 通过 `type` 查询 `alarm_scheme_device_rule` 表
4. 反查告警方案，创建告警实例

#### 委托人处理
- 告警实例中记录委托人信息
- 同步到 ThingsBoard 的 `assigneeId` 字段
- 记录委托操作日志

### 4. 外部接口依赖

## 关键设计点

### 1. 核心映射表
`alarm_scheme_device_rule` 表是整个设计的核心，解决了以下问题：
- ThingsBoard 告警消息反查告警方案
- 告警方案与设备的多对多关系
- ThingsBoard 规则同步状态跟踪

### 2. 联动触发
- 由 ThingsBoard 实现，降低系统复杂性
- 支持当前设备多指标联动
- 支持跨设备指标联动

### 3. 启动规则
- 配置同步到 ThingsBoard
- 由 ThingsBoard 控制告警启用时间
- 支持 4 种启动规则类型

### 4. 委托人机制
- 告警实例记录委托人信息
- 同步到 ThingsBoard
- 支持委托操作日志
