# 告警管理系统详细设计 v2

## 一、核心概念说明

### 1.1 告警方案与ThingsBoard告警规则的关系

```
告警方案（本地）
  ├─ 关联测站（多个）
  │   └─ 测站关联设备（通过接口查询）
  │       └─ 为每个设备在ThingsBoard创建告警规则
  │           └─ 告警规则类型（type）= 告警方案唯一标识
  └─ 指标规则配置
      ├─ 可配置当前设备的多个指标
      └─ 可配置其他设备的指标（联动）
```

### 1.2 告警触发流程

```
1. 用户在告警管理系统创建告警方案
2. 系统为关联的每个设备在ThingsBoard创建告警规则
   - 告警规则类型（type）设置为唯一标识（如：scheme_id + metric_key）
   - 启动规则、阈值条件、联动条件都同步到ThingsBoard
3. ThingsBoard监控设备数据，满足条件时触发告警
4. ThingsBoard通过Kafka发送告警消息（包含type字段）
5. 告警管理系统消费消息，通过type反查告警方案
6. 创建告警实例，发送通知
```

### 1.3 ThingsBoard告警消息结构

基于 `thingsboard-alert.json` 的消息结构：
```json
{
  "id": {"entityType": "ALARM", "id": "uuid"},
  "type": "电量不足",
  "originator": {"entityType": "DEVICE", "id": "device_uuid"},
  "severity": "WARNING",
  "status": "ACTIVE_UNACK",
  "acknowledged": false,
  "cleared": false,
  "assigneeId": null,
  "startTs": 1770301055898,
  "endTs": 1772469953511,
  "originatorName": "喜盈门地下1号井",
  "details": {}
}
```

关键字段：
- `type`: 告警规则类型，用于反查告警方案
- `originator.id`: ThingsBoard设备ID
- `severity`: 告警级别
- `status`: 告警状态
- `assigneeId`: 委托人ID


## 二、数据库设计

### 2.1 告警方案相关表

#### 2.1.1 告警方案表 (alarm_scheme)

```sql
CREATE TABLE alarm_scheme (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
  scheme_code VARCHAR(50) NOT NULL COMMENT '告警方案编码（唯一）',
  scheme_name VARCHAR(100) NOT NULL COMMENT '告警方案名称',
  station_type_id BIGINT NOT NULL COMMENT '测站类型ID',
  alarm_message VARCHAR(500) COMMENT '告警提示语',
  enable_rule_type VARCHAR(20) NOT NULL DEFAULT 'ALWAYS' COMMENT '启动规则类型：ALWAYS-始终启动,SCHEDULED-定时启动,CUSTOM-自定义时间,CONDITION-工况启动',
  enable_rule_config JSON COMMENT '启动规则配置JSON',
  status TINYINT NOT NULL DEFAULT 1 COMMENT '状态：0-禁用,1-启用',
  created_by BIGINT COMMENT '创建人ID',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_by BIGINT COMMENT '更新人ID',
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  deleted TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除：0-未删除,1-已删除',
  UNIQUE KEY uk_code (scheme_code),
  INDEX idx_station_type (station_type_id),
  INDEX idx_status (status, deleted),
  INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警方案表';
```

**字段说明**：
- `scheme_code`: 告警方案唯一编码，用于生成ThingsBoard告警规则类型（type）

#### 2.1.2 告警方案测站关联表 (alarm_scheme_station)

```sql
CREATE TABLE alarm_scheme_station (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
  scheme_id BIGINT NOT NULL COMMENT '告警方案ID',
  station_id BIGINT NOT NULL COMMENT '测站ID（外部系统）',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  UNIQUE KEY uk_scheme_station (scheme_id, station_id),
  INDEX idx_station (station_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警方案测站关联表';
```

#### 2.1.3 告警方案指标规则表 (alarm_scheme_metric_rule)

```sql
CREATE TABLE alarm_scheme_metric_rule (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
  scheme_id BIGINT NOT NULL COMMENT '告警方案ID',
  metric_key VARCHAR(100) NOT NULL COMMENT '监测指标键',
  metric_name VARCHAR(100) NOT NULL COMMENT '监测指标名称',
  metric_alias VARCHAR(100) COMMENT '监测指标别名',
  device_id BIGINT COMMENT '设备ID（为空表示当前设备，不为空表示其他设备-用于联动）',
  trigger_type VARCHAR(20) NOT NULL DEFAULT 'INDEPENDENT' COMMENT '触发类型：INDEPENDENT-独立触发,LINKAGE-联动触发',
  linkage_group_id BIGINT COMMENT '联动组ID，同一组内的规则需同时满足',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  INDEX idx_scheme (scheme_id),
  INDEX idx_linkage_group (linkage_group_id),
  INDEX idx_device (device_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警方案指标规则表';
```

**字段说明**：
- `device_id`: 为空表示当前设备的指标，不为空表示其他设备的指标（用于联动触发）
- `linkage_group_id`: 联动组ID，同一组内的所有规则需同时满足才触发告警

#### 2.1.4 告警级别条件表 (alarm_level_condition)

```sql
CREATE TABLE alarm_level_condition (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
  metric_rule_id BIGINT NOT NULL COMMENT '指标规则ID',
  alarm_level VARCHAR(20) NOT NULL COMMENT '告警级别：CRITICAL-危险,MAJOR-重要,MINOR-次要,WARNING-警告,INDETERMINATE-轻微',
  condition_operator VARCHAR(20) NOT NULL COMMENT '条件操作符：GT-大于,GTE-大于等于,LT-小于,LTE-小于等于,EQ-等于,NEQ-不等于,BETWEEN-区间',
  threshold_value DECIMAL(20,6) COMMENT '阈值',
  threshold_min DECIMAL(20,6) COMMENT '区间最小值',
  threshold_max DECIMAL(20,6) COMMENT '区间最大值',
  duration_seconds INT COMMENT '持续时间(秒)',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  INDEX idx_metric_rule (metric_rule_id),
  INDEX idx_alarm_level (alarm_level)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警级别条件表';
```

#### 2.1.5 告警方案设备规则映射表 (alarm_scheme_device_rule)

```sql

  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
  scheme_id BIGINT NOT NULL COMMENT '告警方案ID',
  device_id BIGINT NOT NULL COMMENT '设备ID（外部系统）',
  tb_device_id VARCHAR(100) NOT NULL COMMENT 'ThingsBoard设备ID',
  tb_alarm_type VARCHAR(100) NOT NULL COMMENT 'ThingsBoard告警规则类型（唯一标识）',

  sync_status VARCHAR(20) NOT NULL DEFAULT 'PENDING' COMMENT '同步状态：PENDING-待同步,SUCCESS-成功,FAILED-失败',
  sync_time DATETIME COMMENT '同步时间',
  sync_error VARCHAR(500) COMMENT '同步失败原因',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  UNIQUE KEY uk_tb_alarm_type (tb_alarm_type),
  INDEX idx_scheme (scheme_id),
  INDEX idx_device (device_id),
  INDEX idx_tb_device (tb_device_id),
  INDEX idx_sync_status (sync_status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='告警方案设备规则映射表';
```

**字段说明**：
- `tb_alarm_type`: ThingsBoard告警规则类型，格式建议：`{scheme_code}_{device_id}`，用于反查告警方案
- `tb_alarm_rule_id`: ThingsBoard返回的告警规则ID
- `sync_status`: 同步状态，用于跟踪规则是否成功同步到ThingsBoard

**设计说明**：
这张表是核心映射表，解决了通过ThingsBoard告警消息中的`type`字段反查告警方案的问题。

## 三、核心业务流程设计

### 3.1 告警方案创建与同步流程

```
1. 用户创建告警方案
   ├─ 输入：方案名称、测站类型、关联测站、指标规则、启动规则等
   └─ 生成唯一的 scheme_code

2. 保存告警方案到本地数据库
   ├─ alarm_scheme: 方案主表
   ├─ alarm_scheme_station: 测站关联
   ├─ alarm_scheme_metric_rule: 指标规则
   └─ alarm_level_condition: 级别条件

3. 查询测站关联的设备（调用外部接口）
   └─ 获取设备列表及ThingsBoard设备ID

4. 为每个设备在ThingsBoard创建告警规则
   ├─ 生成 tb_alarm_type = {scheme_code}_{device_id}
   ├─ 构建告警规则配置（包含指标、阈值、启动规则、联动条件）
   ├─ 调用ThingsBoard API创建告警规则
   └─ 保存映射关系到 alarm_scheme_device_rule 表

5. 同步状态跟踪
   ├─ 成功：sync_status = SUCCESS, 记录 tb_alarm_rule_id
   └─ 失败：sync_status = FAILED, 记录 sync_error
```

### 3.2 告警触发与处理流程

```
1. ThingsBoard监控设备数据
   └─ 满足告警条件时触发告警

2. ThingsBoard通过Kafka发送告警消息
   └─ 消息包含：tb_alarm_id, type, originator, severity, status等

3. 告警管理系统消费Kafka消息
   ├─ 解析消息获取 type 字段
   └─ 通过 type 查询 alarm_scheme_device_rule 表

4. 反查告警方案
   ├─ 根据 tb_alarm_type 查询 alarm_scheme_device_rule
   ├─ 获取 scheme_id 和 device_id
   └─ 查询完整的告警方案配置

5. 查询设备关联的测站（调用外部接口）
   └─ 获取测站列表

6. 创建或更新告警实例
   ├─ 检查是否已存在（根据 tb_alarm_id）
   ├─ 如果不存在：创建新告警实例
   │   ├─ alarm_instance: 告警主表
   │   ├─ alarm_instance_station: 测站关联
   │   └─ alarm_status_history: 状态历史
   └─ 如果已存在：更新告警实例
       ├─ 更新 trigger_count、last_trigger_time
       ├─ 更新 alarm_status（根据ThingsBoard消息）
       └─ 记录状态变更历史

7. 发送告警通知消息到Kafka
   └─ Topic: alarm.notification.events

8. 通知模块消费消息
   ├─ 匹配通知规则
   ├─ 渲染通知模板
   ├─ 查询接收用户组
   └─ 发送通知任务到Kafka
```

### 3.3 告警方案更新与同步流程

```
1. 用户更新告警方案
   └─ 修改指标规则、阈值、启动规则等

2. 更新本地数据库
   └─ 更新相关表记录

3. 查询该方案关联的所有设备规则
   └─ 从 alarm_scheme_device_rule 表查询

4. 批量更新ThingsBoard告警规则
   ├─ 调用ThingsBoard API更新每个设备的告警规则
   └─ 更新 sync_status 和 sync_time

5. 处理同步失败的规则
   └─ 记录失败原因，支持重试
```

### 3.4 告警方案删除流程

```
1. 用户删除告警方案（软删除）
   └─ 设置 deleted = 1

2. 查询该方案关联的所有设备规则
   └─ 从 alarm_scheme_device_rule 表查询

3. 批量删除ThingsBoard告警规则
   ├─ 调用ThingsBoard API删除每个设备的告警规则
   └─ 删除 alarm_scheme_device_rule 记录

4. 处理活跃告警
   └─ 已存在的告警实例保留，但不再产生新告警
```

## 四、关键技术点说明

### 4.1 ThingsBoard告警规则类型（type）生成策略

**方案一：scheme_code + device_id**
```
格式：{scheme_code}_{device_id}
示例：PUMP_TEMP_001_3001
优点：唯一性强，便于反查
缺点：较长
```

**方案二：scheme_id + device_id**
```
格式：SCHEME_{scheme_id}_DEV_{device_id}
示例：SCHEME_1001_DEV_3001
优点：简洁，唯一性强
缺点：不够语义化
```

**推荐方案一**，因为 scheme_code 可以设置为有意义的编码，便于在ThingsBoard中识别。

### 4.2 联动触发的实现

由于ThingsBoard支持联动触发（一个告警规则可以同时设置多个指标），联动逻辑在ThingsBoard层面实现：

**在创建ThingsBoard告警规则时**：
```json
{
  "alarmType": "PUMP_TEMP_001_3001",
  "alarmDetails": {
    "conditions": [
      {
        "deviceId": "current",
        "metricKey": "temperature",
        "operator": "GT",
        "threshold": 80
      },
      {
        "deviceId": "other_device_uuid",
        "metricKey": "pressure",
        "operator": "GT",
        "threshold": 150
      }
    ],
    "linkageMode": "ALL"
  }
}
```

**告警管理系统的处理**：
- 在 `alarm_scheme_metric_rule` 表中记录所有指标规则
- `device_id` 字段区分当前设备和其他设备
- `linkage_group_id` 标识哪些指标需要联动
- 同步到ThingsBoard时，将同一 linkage_group_id 的规则组合成联动条件

### 4.3 启动规则的同步

启动规则配置同步到ThingsBoard，由ThingsBoard控制告警的启用时间：

**启动规则类型**：
1. **ALWAYS（始终启动）**：告警规则始终有效
2. **SCHEDULED（定时启动）**：指定开始和结束时间
3. **CUSTOM（自定义时间）**：指定星期几和时间段
4. **CONDITION（工况启动）**：根据其他条件判断（预留）

**同步到ThingsBoard**：
```json
{
  "alarmType": "PUMP_TEMP_001_3001",
  "schedule": {
    "type": "CUSTOM",
    "weekDays": [1, 2, 3, 4, 5],
    "timeRanges": [
      {"start": "08:00", "end": "12:00"},
      {"start": "14:00", "end": "18:00"}
    ]
  }
}
```

### 4.4 告警实例的创建与更新逻辑

**创建告警实例**（首次触发）：
```
1. 检查 tb_alarm_id 是否已存在
2. 如果不存在，创建新记录
   - 设置 first_trigger_time = 当前时间
   - 设置 last_trigger_time = 当前时间
   - 设置 trigger_count = 1
   - 设置 alarm_status = ACTIVE
3. 创建测站关联记录（alarm_instance_station）
4. 记录状态历史（alarm_status_history）
```

**更新告警实例**（重复触发）：
```
1. 根据 tb_alarm_id 查询已存在的告警实例
2. 更新字段
   - trigger_count += 1
   - last_trigger_time = 当前时间
   - 如果级别变化，更新 alarm_level
   - 如果状态变化，更新 alarm_status
3. 如果状态变化，记录状态历史
```

**处理ThingsBoard的状态变化**：
```
ThingsBoard状态 -> 告警管理系统状态映射：
- ACTIVE_UNACK -> ACTIVE（报警中）
- ACTIVE_ACK -> ACKNOWLEDGED（已确认）
- CLEARED_UNACK -> CLEARED（已清除）
- CLEARED_ACK -> CLEARED（已清除）
```

### 4.5 委托人（assignee）的处理

**委托流程**：
```
1. 用户在告警详情页点击"委托"按钮
2. 选择委托人
3. 调用接口更新告警实例
   - 设置 assignee_id 和 assignee_name
   - 同步到ThingsBoard（调用API更新 assigneeId）
4. 记录处理日志（handle_type = ASSIGN）
5. 发送通知给委托人
```

**ThingsBoard同步**：
```
调用ThingsBoard API更新告警的 assigneeId 字段，
保持ThingsBoard和告警管理系统的委托人信息一致。
```

## 七、数据字典

### 7.1 告警级别（alarm_level）

| 代码 | 名称 | 优先级 | 说明 |
|------|------|--------|------|
| CRITICAL | 危险 | 1 | 最高级别，需要立即处理 |
| MAJOR | 重要 | 2 | 重要告警，需要尽快处理 |
| MINOR | 次要 | 3 | 次要告警，需要关注 |
| WARNING | 警告 | 4 | 警告级别，需要注意 |
| INDETERMINATE | 轻微 | 5 | 轻微告警，可延后处理 |

### 7.2 告警状态（alarm_status）

| 代码 | 名称 | 说明 |
|------|------|------|
| ACTIVE | 报警中 | 告警处于活跃状态 |
| ACKNOWLEDGED | 已确认 | 告警已被确认 |
| CLEARED | 已清除 | 告警已清除（条件不再满足） |
| RESOLVED | 已处置 | 告警已处置完成 |

### 7.3 触发类型（trigger_type）

| 代码 | 名称 | 说明 |
|------|------|------|
| INDEPENDENT | 独立触发 | 单个指标满足条件即触发 |
| LINKAGE | 联动触发 | 多个指标同时满足条件才触发 |

### 7.4 启动规则类型（enable_rule_type）

| 代码 | 名称 | 说明 |
|------|------|------|
| ALWAYS | 始终启动 | 告警规则始终有效 |
| SCHEDULED | 定时启动 | 指定开始和结束时间 |
| CUSTOM | 自定义时间启动 | 指定星期几和时间段 |
| CONDITION | 工况启动 | 根据其他条件判断（预留） |

### 7.5 条件操作符（condition_operator）

| 代码 | 名称 | 符号 | 说明 |
|------|------|------|------|
| GT | 大于 | > | 值大于阈值 |
| GTE | 大于等于 | >= | 值大于等于阈值 |
| LT | 小于 | < | 值小于阈值 |
| LTE | 小于等于 | <= | 值小于等于阈值 |
| EQ | 等于 | = | 值等于阈值 |
| NEQ | 不等于 | != | 值不等于阈值 |
| BETWEEN | 区间 | between | 值在区间范围内 |

### 7.6 通知类型（notify_type）

| 代码 | 名称 | 说明 |
|------|------|------|
| BEHAVIOR | 行为通知 | 设备行为变化通知（上线、离线等） |
| ALARM | 报警通知 | 告警事件通知 |

### 7.7 通知渠道（notify_channel）

| 代码 | 名称 | 说明 |
|------|------|------|
| WEB | 站内消息 | 系统内部消息通知 |
| EMAIL | 邮件 | 邮件通知 |
| SMS | 短信 | 短信通知 |

### 7.8 处理类型（handle_type）

| 代码 | 名称 | 说明 |
|------|------|------|
| ACKNOWLEDGE | 确认 | 确认告警 |
| ASSIGN | 委托 | 委托他人处理 |
| HANDLE | 处置 | 处置告警 |
| CLEAR | 清除 | 清除告警 |
| COMMENT | 评论 | 添加评论 |

## 八、核心业务场景示例

### 8.1 场景一：创建告警方案并同步到ThingsBoard

**业务场景**：
创建一个泵站高温告警方案，监控温度指标，当温度超过80℃时触发危险级别告警。

**操作步骤**：

1. **创建告警方案**
```
告警方案信息：
- scheme_code: PUMP_TEMP_001
- scheme_name: 泵站高温规则
- station_type_id: 1001（管网液位测站）
- alarm_message: 温度超过阈值，请及时处理
- enable_rule_type: CUSTOM
- enable_rule_config: {"weekDays":[1,2,3,4,5],"timeRanges":[{"start":"08:00","end":"18:00"}]}
```

2. **关联测站**
```
关联测站：
- station_id: 1001（泵站A）
- station_id: 1002（泵站B）
```

3. **配置指标规则**
```
指标规则：
- metric_key: temperature
- metric_name: 温度
- trigger_type: INDEPENDENT
- 级别条件：
  - CRITICAL: temperature > 80
  - MAJOR: temperature > 70
  - WARNING: temperature > 60
```

4. **查询测站关联的设备**
```
调用外部接口：
GET /api/stations/1001/devices
GET /api/stations/1002/devices

获取设备列表：
- device_id: 3001, tb_device_id: uuid_001
- device_id: 3002, tb_device_id: uuid_002
- device_id: 3003, tb_device_id: uuid_003
```

5. **为每个设备创建ThingsBoard告警规则**
```
设备3001：
- tb_alarm_type: PUMP_TEMP_001_3001
- 调用ThingsBoard API创建告警规则
- 保存映射关系到 alarm_scheme_device_rule

设备3002：
- tb_alarm_type: PUMP_TEMP_001_3002
- 调用ThingsBoard API创建告警规则
- 保存映射关系到 alarm_scheme_device_rule

设备3003：
- tb_alarm_type: PUMP_TEMP_001_3003
- 调用ThingsBoard API创建告警规则
- 保存映射关系到 alarm_scheme_device_rule
```

### 8.2 场景二：ThingsBoard触发告警并创建告警实例

**业务场景**：
设备3001的温度达到85℃，ThingsBoard触发告警。

**处理流程**：

1. **ThingsBoard触发告警**
```

```

2. **ThingsBoard发送Kafka消息**
```json
{
  "id": {"entityType": "ALARM", "id": "alarm_uuid_123"},
  "type": "PUMP_TEMP_001_3001",
  "originator": {"entityType": "DEVICE", "id": "uuid_001"},

  "status": "ACTIVE_UNACK",
  "startTs": 1709452800000,

4. **查询告警方案详情**
```

```

5. **查询设备关联的测站**
```
调用外部接口：

## 九、异常场景处理

### 9.1 ThingsBoard同步失败

**场景**：创建告警方案后，同步到ThingsBoard失败。

**处理方案**：
```
1. 记录同步失败状态
   - alarm_scheme_device_rule.sync_status = FAILED
   - alarm_scheme_device_rule.sync_error = 失败原因

2. 提供重试机制
   - 定时任务扫描 sync_status = FAILED 的记录
   - 重新调用ThingsBoard API
   - 更新同步状态

3. 告警提示
   - 在告警方案列表显示同步状态
   - 同步失败的方案标记为异常状态
```

### 9.2 外部接口调用失败

**场景**：查询设备关联的测站时，外部接口调用失败。

**处理方案**：
```
1. 重试机制
   - 设置重试次数（如3次）
   - 指数退避策略

2. 降级处理
   - 如果查询测站失败，告警实例仍然创建
   - alarm_instance_station 表暂不插入数据
   - 后续通过补偿任务补充测站信息

3. 日志记录
   - 记录接口调用失败日志
   - 便于排查问题
```

### 9.3 Kafka消息消费失败

**场景**：消费ThingsBoard告警消息时发生异常。

**处理方案**：
```
1. 异常捕获
   - 捕获消费异常，记录日志
   - 不影响其他消息的消费

2. 死信队列
   - 将处理失败的消息发送到死信队列
   - 后续人工介入处理

3. 消息重试
   - 对于临时性错误（如数据库连接失败），支持重试
   - 设置最大重试次数
```

### 9.4 告警方案删除后仍收到告警

**场景**：删除告警方案后，ThingsBoard仍然触发告警。

**处理方案**：
```
1. 检查告警方案状态
   - 消费消息时，检查 alarm_scheme.deleted 字段
   - 如果已删除，忽略该告警消息

2. 清理ThingsBoard规则
   - 删除告警方案时，确保删除所有ThingsBoard告警规则
   - 如果删除失败，记录失败状态，支持重试

3. 告警实例处理
   - 已存在的告警实例保留
   - 不再创建新的告警实例
```

### 9.5 设备从测站移除后的告警

**场景**：设备从测站移除后，该设备仍然触发告警。

**处理方案**：
```
1. 监听设备-测站关联变化
   - 当设备从测站移除时，触发事件
   - 删除对应的ThingsBoard告警规则

2. 告警实例处理
   - 已存在的告警实例保留
   - 标记为"设备已移除"状态

3. 定期清理
   - 定时任务清理无效的告警规则
   - 同步设备-测站关联关系
```

## 十、性能优化建议

### 10.1 数据库优化

**索引优化**：
```sql
-- 告警实例查询优化
CREATE INDEX idx_alarm_query ON alarm_instance(alarm_status, alarm_level, first_trigger_time);

-- 告警方案设备规则查询优化
CREATE INDEX idx_tb_alarm_type ON alarm_scheme_device_rule(tb_alarm_type);

-- 通知记录查询优化
CREATE INDEX idx_notify_query ON notify_record(notify_type, send_status, created_at);
```

**分区表**：
```sql
-- 告警实例按月分区
ALTER TABLE alarm_instance PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
  PARTITION p202601 VALUES LESS THAN (202602),
  PARTITION p202602 VALUES LESS THAN (202603),
  ...
);

-- 通知记录按月分区
ALTER TABLE notify_record PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
  PARTITION p202601 VALUES LESS THAN (202602),
  PARTITION p202602 VALUES LESS THAN (202603),
  ...
);
```

### 10.2 缓存优化

**告警方案缓存**：
```
- 缓存告警方案配置（scheme_id -> 方案详情）
- 缓存设备规则映射（tb_alarm_type -> scheme_id, device_id）
- 缓存过期时间：1小时
- 更新告警方案时，清除缓存
```

**设备信息缓存**：
```
- 缓存设备基本信息（device_id -> 设备详情）
- 缓存设备-测站关联（device_id -> 测站列表）
- 缓存过期时间：30分钟
```

### 10.3 批量处理优化

**批量同步ThingsBoard规则**：
```
- 创建告警方案时，批量创建ThingsBoard规则
- 使用线程池并发调用ThingsBoard API
- 设置合理的并发数（如10个线程）
```

**批量查询优化**：
```
- 查询告警列表时，批量查询关联数据
- 使用 IN 查询代替多次单条查询
- 减少数据库往返次数
```

### 10.4 Kafka消费优化

**并发消费**：
```
- 设置合理的消费者线程数
- 根据分区数配置消费者数量
- 避免单线程消费成为瓶颈
```

**批量提交**：
```
- 批量消费消息后统一提交offset
- 减少提交频率，提升性能
- 注意消息重复消费的处理
```
