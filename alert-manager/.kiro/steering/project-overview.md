# 告警管理系统项目概述

## 项目背景

本项目是物联汇聚应用平台中的告警管理子系统，基于 ThingsBoard 作为基座开发。平台包含设备管理系统、测站管理系统、数据处理系统和告警管理系统等多个子系统。

## 核心目标

- 提供灵活的告警方案配置能力
- 支持独立触发和联动触发两种告警模式
- 支持多种启动规则（始终、定时、自定义时间、工况）
- 实现告警的全生命周期管理（创建、确认、委托、处置、清除）
- 提供多渠道通知能力（WEB、邮件、短信）

## 技术栈规范

### 后端技术栈
- **JDK**: 17+
- **框架**: Spring Boot 3.X
- **ORM**: MyBatis-Plus
- **消息队列**: MQTT
- **数据库**: PostgreSQL 14+
- **缓存**: Redis 6.0+（可选）
- **基座**: ThingsBoard

### 编码规范
- 遵循阿里巴巴 Java 开发手册
- 使用 Lombok 简化代码
- 统一使用 RESTful API 风格

## 系统架构

### 核心模块
1. **告警方案管理模块**: 告警方案的 CRUD、同步到 ThingsBoard
2. **告警处理模块**: 消费 ThingsBoard 告警消息、创建告警实例
3. **通知管理模块**: 通知规则匹配、通知发送
4. **统计分析模块**: 告警统计、Dashboard 数据

### 数据流向
```
ThingsBoard 规则链 
  → MQTT (tb.alarm.events)
  → 告警管理系统消费
  → 创建/更新告警实例
  → 通知管理系统
  → 多渠道通知发送
```

## 关键设计原则

1. **职责分离**: ThingsBoard 负责告警触发，告警管理系统负责业务处理
2. **异步解耦**: 使用 MQTT 实现模块间异步通信
3. **配置驱动**: 告警规则配置化，支持灵活调整
4. **可扩展性**: 预留扩展点，支持后续功能增强
5. **高可用**: 支持分布式部署，无单点故障

## 核心概念

### 告警方案（Alarm Scheme）
- 告警方案是告警规则的集合
- 一个告警方案关联多个测站
- 测站关联多个设备
- 为每个设备在 ThingsBoard 创建告警规则

### ThingsBoard 告警规则类型（tb_alarm_type）
- 格式: `{scheme_code}_{device_id}`
- 作用: 作为唯一标识，用于反查告警方案
- 示例: `PUMP_TEMP_001_3001`

### 联动触发
- 由 ThingsBoard 实现联动逻辑
- 支持当前设备多指标联动
- 支持跨设备指标联动

### 启动规则
- 配置同步到 ThingsBoard
- 由 ThingsBoard 控制告警启用时间
- 支持 4 种类型: ALWAYS、SCHEDULED、CUSTOM、CONDITION
